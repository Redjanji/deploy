## 5. 部署前置条件

### 5.1 两种部署方式对比

| 方式 | 适用场景 | 需要上传的内容 | 运维复杂度 |
|------|----------|---------------|-----------|
| **方式 A：预构建镜像（推荐）** | 生产/测试环境，交给运维 | `deploy/` 目录 + 镜像 tar.gz 包（含全部 17 个镜像） | ⭐ 最简单 |
| 方式 B：源码构建 | 开发环境，经常改代码 | 全部项目源码 | ⭐⭐⭐ |

> **本文档默认使用方式 A（预构建镜像）**。方式 B 详见附录 D。

### 5.2 Docker 环境准备

**如果服务器已有 Docker 24.0+，可跳过此节。**

#### 5.2.1 安装 Docker（Ubuntu，推荐阿里云源）

```bash
# 1. 安装前置依赖
sudo apt update
sudo apt install -y ca-certificates curl gnupg lsb-release

# 2. 添加阿里云 Docker GPG 密钥
curl -fsSL https://mirrors.aliyun.com/docker-ce/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg

# 3. 添加阿里云 Docker 软件源
echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/docker-archive-keyring.gpg] https://mirrors.aliyun.com/docker-ce/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# 4. 安装 Docker
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# 5. 设置开机自启
sudo systemctl enable docker
sudo systemctl start docker

# 6. 将当前用户加入 docker 组（免 sudo）
sudo usermod -aG docker $USER
newgrp docker

# 7. 验证安装
docker --version
docker compose version
```

#### 5.2.2 配置 Docker 镜像加速器（可选，国内服务器推荐）

> 注意：本项目为离线部署，镜像已打包在 tar.gz 中，不需要从 Docker Hub 拉取。
> 如果后续需要额外拉取镜像，建议配置加速器。

```bash
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json << 'EOF'
{
  "registry-mirrors": [
    "https://docker.mirrors.ustc.edu.cn",
    "https://hub-mirror.c.163.com"
  ]
}
EOF
sudo systemctl daemon-reload
sudo systemctl restart docker
```

## 6. 部署操作步骤（方式 A：预构建分片镜像）

### 6.1 总体流程

```
开发机：构建镜像 → 分片导出（50MB/片，Git可上传）
    ↓
Git：分片文件提交到仓库（每片 <100MB）
    ↓
服务器：git clone 拉取 → 一键部署脚本自动合并+启动
```

**一键部署（推荐）**：

```bash
chmod +x full-deploy.sh && ./full-deploy.sh
```

### 6.2 【开发机】步骤一：构建镜像

```powershell
# 在项目根目录执行
cd one-click-deployment/deploy

# 构建所有 11 个服务的镜像（首次约 15-30 分钟）
# 使用项目根目录的 Dockerfile 构建各服务镜像
```

> **说明**：Dockerfile 采用多阶段构建，builder 阶段使用 `eclipse-temurin:21-jdk-jammy` + Maven 编译，运行阶段使用 `eclipse-temurin:21-jre-alpine` 精简镜像。所有服务共享同一个 Dockerfile。

### 6.3 【开发机】步骤二：分片导出镜像

使用 PowerShell 分片（每片 50MB，确保 Git 可上传）：

```powershell
# 在 deploy 目录执行
$file = "xss-images-20260716.tar"
$chunkSize = 50MB
$chunkIndex = 1
Get-Content $file -Encoding Byte -ReadCount $chunkSize | ForEach-Object {
    $chunkName = "$file.part{0:D2}" -f $chunkIndex
    Set-Content $chunkName -Value $_ -Encoding Byte
    $chunkIndex++
}
```

执行成功后，会在 `deploy/` 目录生成：
```
xss-images-20260716.tar.part01  (50 MB)
xss-images-20260716.tar.part02  (50 MB)
...
xss-images-20260716.tar.part12  (~18 MB)
```

> **总计约 12 片，共 ~570MB，每片 <100MB，可正常 Git 提交**

> **注意**：原 tar 文件可以删除，只保留 part 分片文件即可。Git 提交时将所有 part 文件一起提交。

### 6.4 【服务器】步骤三：拉取部署包

```bash
# Git 拉取项目
git clone <仓库地址> /opt/xss
cd /opt/xss/one-click-deployment/deploy

# 确认分片文件齐全
ls xss-images-*.tar.part*
# 应看到 part01 ~ part12
```

### 6.5 【服务器】步骤四：配置环境变量

```bash
cd /opt/xss/one-click-deployment/deploy

# 复制模板
cp .env.example .env

# 编辑 .env，修改以下关键项（生产环境必须修改）
vi .env
```

**必须修改的配置项**：

```ini
# MySQL root 密码（强密码）
MYSQL_ROOT_PASSWORD=<你的强密码>

# Redis 密码
REDIS_PASSWORD=<你的强密码>

# RabbitMQ 凭据（不要使用 guest）
RABBITMQ_USER=xss_admin
RABBITMQ_PASS=<你的强密码>

# MinIO 凭据
MINIO_ACCESS_KEY=<你的AccessKey>
MINIO_SECRET_KEY=<你的SecretKey>

# JWT 密钥（使用 openssl rand -base64 32 生成）
JWT_SECRET=<生成的Base64密钥>

# 应用凭证密钥
```

## 12. 常见问题排查

### 12.1 服务启动失败

**现象**：某服务容器启动后立即退出。

**排查步骤**：

```bash
# 1. 查看退出日志
docker compose logs <service-name>

# 2. 常见原因
# - Nacos 未就绪 → 等待 nacos healthy 后重启
# - 数据库未就绪 → 等待 mysql healthy 后重启
# - 配置缺失 → 检查 Nacos 配置是否完整
# - 端口冲突 → 检查宿主机端口占用
```

### 12.2 Nacos 服务未注册

**现象**：服务启动但 Nacos 控制台看不到实例。

**排查**：

```bash
# 1. 检查服务是否能连接 Nacos
docker exec xss-booking curl -s http://nacos:8848/nacos/v1/console/health/readiness

# 2. 检查环境变量
docker exec xss-booking env | grep NACOS

# 3. 查看服务日志中的注册信息
docker compose logs booking-service | grep -i nacos
```

### 12.3 数据库连接失败

**现象**：服务日志报 `Communications link failure`。

**排查**：

```bash
# 1. 检查 MySQL 容器状态
docker compose ps mysql

# 2. 从业务容器内测试连接
docker exec xss-booking curl -s telnet://mysql:3306
# 或
docker exec xss-booking sh -c "echo 'SELECT 1' | mysql -h mysql -uroot -p$MYSQL_PASSWORD"

# 3. 检查密码是否一致
# .env 中的 MYSQL_ROOT_PASSWORD 必须与 docker-compose.yml 中传递的一致
```

### 12.4 网关返回 503

**现象**：访问网关返回 503 Service Unavailable。

**原因**：目标服务未注册到 Nacos 或已下线。

**排查**：

```bash
# 1. 检查目标服务容器状态
docker compose ps <service-name>

# 2. 检查 Nacos 实例列表
curl "http://localhost:8848/nacos/v1/ns/instance/list?serviceName=<service-name>"

# 3. 重启目标服务
docker compose restart <service-name>
```

### 12.5 内存不足 / 服务器卡死

**现象**：部署过程中服务器无响应，或容器频繁 OOM 退出（exit code 137）。

**原因**：所有服务同时启动，瞬时内存占用过高，触发 OOM Killer。

**解决方案**：

1. **使用分批启动（推荐）**：
   ```bash
   # 第 1 批：中间件
   docker compose --profile infra up -d
   # 等待全部健康后
   docker compose --profile core up -d
   # 等待核心服务就绪后
   docker compose --profile business up -d
   docker compose --profile nginx up -d
   ```

2. **降低单个服务内存限制**（在 docker-compose.yml 中调整）：
   ```yaml
   deploy:
     resources:
       limits:
         memory: 128M   # 从 160M 降到 128M
   ```

3. **降低 JVM 堆内存**（环境变量）：
   ```
   JAVA_TOOL_OPTIONS=-Xms32m -Xmx64m
   ```

4. **检查 OOM**：
   ```bash
   # 查看被 OOM Killer 杀掉的容器
   docker ps -a --filter 'exited=137'
   
   # 查看系统内存
   free -h
   
   # 查看容器资源占用
   docker stats --no-stream
   ```

> **注意**：v1.3 版本已优化为 4GB 可运行，11 个服务 + 6 个中间件总占用约 3.2GB。

### 12.6 Elasticsearch 内存不足

**现象**：ES 容器频繁重启或 OOM。

**解决**：

```bash
# 修改 docker-compose.yml 中 ES 的 JVM 内存
environment:
  - ES_JAVA_OPTS=-Xms1g -Xmx1g  # 从 512m 提升到 1g

# 或降低内存占用
environment:
  - ES_JAVA_OPTS=-Xms256m -Xmx256m
```

### 12.6 镜像构建失败（Maven 下载慢）

**现象**：`docker compose build` 时 Maven 下载依赖超时。

**解决**：在 Dockerfile builder 阶段配置国内 Maven 镜像源：

```dockerfile
# 在 Dockerfile 的 builder 阶段添加
RUN mkdir -p /root/.m2 && cat > /root/.m2/settings.xml <<'EOF'
<settings>
  <mirrors>
    <mirror>
      <id>aliyun</id>
      <name>Aliyun Maven Mirror</name>
      <url>https://maven.aliyun.com/repository/public</url>
      <mirrorOf>central</mirrorOf>
    </mirror>
  </mirrors>
</settings>
EOF
```

### 12.7 时区问题

所有容器已配置 `TZ=Asia/Shanghai`。如发现时间不对：

```bash
# 检查容器时间
docker exec xss-booking date

# 宿主机时间同步
sudo timedatectl set-timezone Asia/Shanghai
sudo apt-get install -y ntpdate && sudo ntpdate ntp.aliyun.com
```

### 12.8 部署过程常见问题汇总

以下是首次部署时可能遇到的问题及解决方案（已在脚本中修复，供参考）：

| 序号 | 问题 | 原因 | 解决方案 |
|------|------|------|----------|
| 1 | Docker 未安装 / docker-compose-plugin 找不到 | Ubuntu 默认源不含 Docker 官方包 | 参考 5.2 节，使用阿里云源安装 Docker |
| 2 | 下载 Docker GPG 密钥时连接重置 | 国内访问 Docker 官方站点不稳定 | 使用阿里云镜像源下载（已在文档中提供） |
| 3 | 当前用户不在 docker 组，权限被拒 | 未将用户加入 docker 组 | `sudo usermod -aG docker $USER && newgrp docker` |
| 4 | Docker Hub 拉取镜像超时 | 服务器在国内，无法直接访问 Docker Hub | 本项目为离线部署，镜像已打包在 tar.gz 中，无需联网 |
| 5 | 端口 3306 / 6379 被占用 | 宿主机原有 MySQL/Redis 服务 | 停止旧服务，或修改 .env 中的端口配置 |
| 6 | SQL 语法错误：`GRANT ... IDENTIFIED BY` | MySQL 8.0 不再支持该语法 | 已修复：拆分为 `CREATE USER` + `GRANT` |
| 7 | SQL 语法错误：`ADD COLUMN IF NOT EXISTS` | MySQL 8.0 不支持该语法（仅 MariaDB 支持） | 已修复：使用存储过程判断字段是否存在 |
| 8 | 删除表时外键约束冲突 | 子表外键引用父表，导致无法直接 DROP | 已修复：所有 SQL 文件开头添加 `SET FOREIGN_KEY_CHECKS = 0;` |
| 9 | docker-compose version 属性废弃警告 | 新版 Compose 不再需要 version 字段 | 已修复：删除 docker-compose.yml 中的 version 行 |
| 10 | 部署后 CPU/内存飙高 | 多个中间件同时启动，ES 资源需求高 | 已优化：分批启动（profiles）+ 内存限制，4GB 可运行 |
| 11 | Git 无法上传大镜像包 | 单文件 > 100MB 被 Git 拒绝 | 已优化：镜像分片为 50MB/片，共约 12 片，可正常提交 |
| 12 | MySQL 容器 OOM 退出（exit code 137）执行 02-dict_db.sql 时 | MySQL 内存限制 384M 不足以执行大 SQL 文件 | 已修复（P001）：MySQL 内存增至 512M，innodb_buffer_pool_size 降至 128M，innodb_log_file_size 降至 32M |
| 13 | init-db.sh 执行失败后不重试，MySQL 重启瞬间连接失败 | SQL 执行时 MySQL 尚未完全稳定 | 已修复（P002）：添加 MySQL 就绪检查（连续 2 次成功）+ 每个 SQL 文件 3 次重试 |
| 14 | 部署后数据库为空（9 个业务数据库未创建） | full-deploy.sh 未自动调用 init-db.sh | 已修复（P003）：full-deploy.sh Step 4 检查数据库完整性，不足 9 个时自动执行 init-db.sh |
| 15 | 前端跨域请求失败：CORS 预检 OPTIONS 被阻止 | Nginx 反向代理未处理 OPTIONS 请求 | 已修复（P004）：nginx.conf 添加 OPTIONS 直接返回 204，包含完整的 CORS 头 |

### 12.9 增量补丁脚本（patch-deploy.sh）

**用途**：对**已有部署**应用增量修复，无需全量重新部署。新部署无需执行此脚本（full-deploy.sh 已包含所有修复）。

**适用场景**：
- 已部署的旧版本需要应用 v1.4 修复
- 想检查本地文件与远程仓库是否有差异
- 仅修复特定问题（如 MySQL OOM、CORS）而不重新部署全部服务

**使用方式**：

```bash
# 1. 仅检查需要修复的项目（不实际修改）
bash patch-deploy.sh --check

# 2. 应用全部补丁（自动备份旧文件、下载最新文件、重启受影响的服务）
bash patch-deploy.sh

# 3. 强制覆盖所有文件（即使本地已是最新）
bash patch-deploy.sh --force
```

**补丁记录**：

| 补丁 ID | 修复内容 | 涉及文件 |
|---------|----------|----------|
| P001 | MySQL 内存 384M→512M, innodb_buffer_pool_size 192M→128M | docker-compose.yml |
| P002 | MySQL 就绪检查（连续 2 次成功）+ 每个 SQL 文件 3 次重试 | init-db.sh |
| P003 | full-deploy.sh 自动检查数据库完整性，不完整时调用 init-db.sh | full-deploy.sh |
| P004 | Nginx CORS OPTIONS 预检请求直接返回 204 | nginx/nginx.conf |
| P005 | Docker 安装使用阿里云镜像源（解决国内 TLS 连接重置） | full-deploy.sh |
| P006 | .env.example 预置正确的 APP_SECRET 配置 | .env.example |
| P007 | docker-compose.yml 中 nacos 服务名统一 | docker-compose.yml |

**补丁脚本工作流程**：

1. **阶段 1 - 检查文件差异**：从 Gitee 下载最新文件，与本地文件 `diff` 比较
2. **阶段 2 - 应用补丁**：备份旧文件（`.bak.<timestamp>`），下载最新文件覆盖
3. **阶段 3 - 重启受影响的服务**：
   - 检测到 `nginx/nginx.conf` 更新 → 重启 Nginx
   - 检测到 `docker-compose.yml` 更新 → 重建 MySQL 容器，等待稳定，检查数据库完整性（不足 9 个时自动执行 init-db.sh）

**注意事项**：
- 补丁脚本需要服务器可访问 Gitee（`gitee.com`）
- 备份文件保留在原目录，文件名格式为 `<原文件名>.bak.<时间戳>`
- 如补丁后服务异常，可手动恢复备份文件：`cp <file>.bak.<timestamp> <file>`

## 13. 备份与恢复

### 13.1 数据备份

```bash
# 备份所有 MySQL 数据库
docker exec xss-mysql mysqldump -uroot -p${MYSQL_ROOT_PASSWORD} \
  --all-databases --single-transaction > backup_$(date +%Y%m%d).sql

# 备份单个数据库
docker exec xss-mysql mysqldump -uroot -p${MYSQL_ROOT_PASSWORD} \
  booking_db > booking_backup_$(date +%Y%m%d).sql

# 备份数据卷
docker run --rm -v mysql-data:/data -v $(pwd):/backup \
  alpine tar czf /backup/mysql-data-backup.tar.gz /data
```

### 13.2 数据恢复

```bash
# 恢复 MySQL
docker exec -i xss-mysql mysql -uroot -p${MYSQL_ROOT_PASSWORD} < backup_20260713.sql

# 恢复数据卷
docker run --rm -v mysql-data:/data -v $(pwd):/backup \
  alpine tar xzf /backup/mysql-data-backup.tar.gz -C /
```

### 13.3 定时备份（推荐）

```bash
# 添加 crontab，每天凌晨 3 点备份
crontab -e
# 添加：
0 3 * * * docker exec xss-mysql mysqldump -uroot -pYOUR_PASSWORD --all-databases > /opt/backups/mysql_$(date +\%Y\%m\%d).sql
```

### 14.2 网络安全检查

- [ ] 仅 8080 端口对外暴露（业务入口）
- [ ] 8848（Nacos）、15672（RabbitMQ）、9001（MinIO）仅内网/VPN 可访问
- [ ] 3306（MySQL）、6379（Redis）、9200（ES）不对外暴露
- [ ] 服务器 SSH 使用密钥登录，禁用 root 远程登录
- [ ] 防火墙已配置，仅开放必要端口

### 14.3 应用安全检查

- [ ] 网关 IP 白名单已启用（生产环境设置 `IP_WHITELIST_ENABLED=true`）
- [ ] IP 白名单列表已配置为允许的客户端 IP
- [ ] HTTPS 已配置（建议通过 Nginx 反向代理终止 TLS）
- [ ] 链路追踪采样率已调整为 10%（生产环境，节省资源）

### 14.4 IP 白名单配置

生产环境强烈建议启用 IP 白名单：

```ini
# .env 中添加
IP_WHITELIST_ENABLED=true
IP_ALLOWED_IPS=127.0.0.1,::1,你的办公网IP/24
```

> **注意**：IP 白名单过滤器位于 JWT 过滤器之前，非白名单 IP 的请求直接返回 403，无法探测认证接口。

## 15. 未来升级路线

### 15.1 当前架构局限性

| 局限 | 说明 | 影响 |
|------|------|------|
| 单机部署 | 所有服务运行在一台机器 | 机器故障即全站宕机 |
| 无集群 | 每个服务仅 1 个实例 | 实例崩溃即服务不可用 |
| 无哨兵 | Redis/MySQL 单点 | 中间件故障即数据不可访问 |
| 手动伸缩 | 无法应对流量突增 | 需人工干预 |

### 15.2 升级路径

**阶段一：多机 Docker Compose（2-3 台机器）**

- 将中间件单独部署到一台机器
- 业务服务部署到另一台机器
- 通过内网通信

**阶段二：Docker Swarm 或 K3s（轻量 K8s）**

- 多台 worker 节点组成集群
- 服务自动调度、故障转移
- 滚动更新、回滚

**阶段三：完整 K8s 集群**

- 使用 K8s Deployment + Service + Ingress
- 配置 HPA（水平 Pod 自动伸缩）
- 引入 Helm Chart 管理部署

**阶段四：中间件高可用**

| 中间件 | 单机方案 | 高可用方案 |
|--------|----------|-----------|
| MySQL | 单实例 | 主从复制 + ProxySQL |
| Redis | 单实例 | Sentinel 哨兵集群 |
| RabbitMQ | 单实例 | 镜像队列集群 |
| Elasticsearch | 单节点 | 多节点集群（最少 3 节点） |
| Nacos | 单机 Derby | 集群 + MySQL 持久化 |

**阶段五：CI/CD**

- Git push 触发自动构建镜像
- 自动化测试
- 滚动部署到测试/生产环境
- 推荐：GitLab CI、GitHub Actions、Jenkins

### 15.3 迁移注意事项

从 Docker Compose 迁移到 K8s 时：

1. **镜像复用**：当前 Dockerfile 生成的镜像可直接用于 K8s
2. **配置迁移**：Nacos 配置无需改动，K8s 中服务仍连接 Nacos
3. **服务发现**：K8s Service 可替代 Nacos 服务发现，或继续使用 Nacos
4. **存储迁移**：数据卷需迁移到 PV/PCS 或对象存储
5. **网络迁移**：Ingress 替代 gateway-service 的对外暴露（但 gateway 仍需保留做业务路由）

将以下脚本保存为 `deploy/start.sh`：

```bash
#!/bin/bash

docker compose up -d

sleep 30

docker compose ps

echo "===== 7. 健康检查 ====="
curl -s http://localhost:8080/actuator/health || echo "⚠️ 网关尚未就绪，请稍后重试"

echo ""
echo "✅ 部署完成！"
echo "   网关地址: http://$(hostname -I | awk '{print $1}'):8080"
echo "   Nacos: http://$(hostname -I | awk '{print $1}'):8848/nacos"

```
